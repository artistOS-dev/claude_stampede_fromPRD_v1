Section 01

Rodeo Status Flow

A rodeo moves through four states. There is no automatic deadline enforcement — each transition is triggered explicitly via API.

pending
Created, entries confirming
voting
Ballots open
closed
Winner computed
archived
Permanent history
TransitionTriggered byWhat happens
pending → voting Target circle accepts challenge Both entries confirmed; voting opens automatically
voting → closed Creator calls POST /api/rodeos/[id]/finalize Borda scores computed, winner stored, credits distributed
closed → archived Creator calls POST /api/rodeos/[id]/archive Moved to permanent circle history — irreversible
⚠️
No automatic close There is no cron job. The rodeo creator must manually call finalize when they decide voting is over, regardless of the end_date field.
Section 02

Casting a Vote

Voters do not pick a single favourite. They submit a ranked ballot — an ordered list of all songs in the rodeo. Position in the list determines the score each song receives.

API

POST /api/rodeos/{id}/rank

// Body — array ordered best → worst
{
  "song_ids": [
    "uuid-song-A",   // rank 1 (favourite)
    "uuid-song-C",   // rank 2
    "uuid-song-B"    // rank 3
  ]
}
  • The submission atomically replaces any previous ballot from that voter.
  • Voters may resubmit as many times as they like while voting is open.
  • Songs omitted from the ballot receive 0 points.

Storage — rodeo_rankings table

create table public.rodeo_rankings (
  id           uuid primary key,
  rodeo_id     uuid,
  voter_id     uuid,
  song_id      uuid,
  rank         integer not null check (rank >= 1),
  submitted_at timestamptz,
  unique(rodeo_id, voter_id, song_id)
);

One row per song per voter. A ballot of 5 songs creates 5 rows, all sharing the same voter_id and rodeo_id.

Section 03

Borda Count Scoring

After all ballots are collected, a Borda count converts ranked preferences into a single numeric score per song.

Formula

For a voter who ranked K songs:

points for rank i  =  K - i + 1

rank 1  →  K points       // highest
rank 2  →  K - 1 points
...
rank K  →  1 point        // lowest
unranked →  0 points

Implementation

// /api/rodeos/[id]/tally — called every ~12 s during voting
const byVoter = new Map()

// group rankings by voter to know each ballot's size K
for (const row of allRankings) {
  byVoter.set(row.voter_id, [...(byVoter.get(row.voter_id) ?? []), row])
}

const songBorda = new Map()

for (const [, ballot] of byVoter) {
  const K = ballot.length                // size of this voter's ballot
  for (const { song_id, rank } of ballot) {
    const pts = K - rank + 1
    const cur = songBorda.get(song_id) ?? { score: 0, ranker_count: 0 }
    cur.score       += pts
    cur.ranker_count += 1
    songBorda.set(song_id, cur)
  }
}

Section 04

Winner Determination

Called inside RodeoService.closeRodeo() when finalize is triggered.

  1. Aggregate per entry. Sum the Borda scores of all songs belonging to each entry (circle or artist).

  2. Select the highest. The entry with the greatest total Borda score is the winner.

  3. Persist. Winner stored in rodeo_results as either winner_circle_id or winner_artist_id. Per-song scores written to rodeo_song_results.weighted_score.

  4. Distribute credits (see Section 05).

  5. Status → closed.

// rodeo-service.ts — entry aggregation
const entryBorda = new Map()   // entryId → total score

for (const [song_id, { score }] of songBorda) {
  const entryId = songToEntry.get(song_id)
  entryBorda.set(entryId, (entryBorda.get(entryId) ?? 0) + score)
}

// pick winner
let winnerId = null, highScore = -1
for (const [entryId, score] of entryBorda) {
  if (score > highScore) { highScore = score; winnerId = entryId }
}
⚠️
Known gap: no tiebreaker If two entries share the identical Borda total, the winner is whichever entry was encountered first during the Map iteration. A deterministic tiebreaker rule has not yet been implemented.

Section 06

Key Rules & Constraints

RuleDetails
Who can vote Subscription tier must be fan, superfan, artist, or producer — free tier cannot vote
Who can open/close Only the rodeo creator (the challenging circle's board/founder)
Who can accept a challenge Board, founder, or admin of the target circle
Song locking Songs are locked after the internal circle vote passes; no substitutions after that
Ballot resubmission Allowed — entire ballot is atomically replaced; voters can refine until voting closes
Archive archived is irreversible — no edits or deletes after this state
Section 07

End-to-End Summary

  1. Challenge issued. Challenging circle's board calls the challenge API. Both circles put in equal credits. Rodeo status: pending.

  2. Challenge accepted. Target circle's board accepts. Entries confirmed for both sides. Status auto-transitions to voting.

  3. Voting period. Members of both circles (and any eligible subscribers) submit ranked ballots via POST /api/rodeos/[id]/rank. Tally refreshes every ~12 seconds.

  4. Finalize. Creator calls POST /api/rodeos/[id]/finalize. RodeoService.closeRodeo() runs.

  5. Borda scores computed. Per-voter K values determined; points assigned to each song; songs aggregated per entry.

  6. Winner selected. Entry with the highest total Borda score declared winner. Results written to rodeo_results and rodeo_song_results.

  7. Credits distributed. Platform fee deducted; remainder split by role (winning artist, songwriter, band, core artists, young bucks, users) and stored in rodeo_credit_distributions.

  8. Status → closed. Results are now visible to all circle members.

  9. Archive. Creator optionally archives the rodeo, moving it permanently to circle history.
